# ADR-001: 単一ホストとローカルSQLiteを正式な出荷構成とする

## Status

Accepted

## Date

2026-07-23

## Context

本アプリケーションはSQLiteを標準データベースとして開発されている。計画中のCSVエクスポートは、同じSQLiteファイル上のスナップショット、所有者付きグローバルロック、監査イベントを利用して、整合性と同時実行数を保証する。

Cloud Runの複数インスタンス、ネットワークファイルシステム、Cloud SQLやSupabase等の外部DBでは、ロックの共有範囲、transaction isolation、ストリーミング配信、一時ファイルの寿命が変わる。同じ保証を検証せずに現在の同期CSV設計を有効化できない。

## Decision

現在の正式な本番出荷構成を、次の条件を満たす単一ホスト構成とする。

- アプリケーションは1台のホスト上で稼働する
- 複数のPHP-FPM workerは同じホスト上で稼働し、同じSQLiteファイルを参照する
- SQLite DBはホストの永続ローカルファイルシステムへ配置する
- SQLiteはWALを使用し、接続ごとのbusy timeoutを5,000ミリ秒とする
- DB、WAL、監査表、一時領域を監視し、DBとWALを整合した方法でバックアップする

Cloud Run等の複数インスタンス、ネットワークファイルシステム、外部DBは、現在の正式な出荷構成に含めない。環境がこの境界に適合しない場合、同期CSVエクスポートはHTTP 503でfail-closedにする。

## Alternatives Considered

### Cloud Runと外部DB

従量課金とマネージド実行環境は魅力だが、現在の同期CSV設計とは排他制御、transaction isolation、一時ファイル配信の前提が異なる。機能保証を維持したまま移行するには、ドライバ別の再設計と再検証が必要なため、今回の出荷構成には採用しない。

### 非同期CSV生成とオブジェクトストレージ配信

複数インスタンスへ拡張しやすく、長時間のHTTP worker占有を避けられる。一方、ジョブ基盤、保存期間、ダウンロード認可、オブジェクト削除を追加設計する必要があるため、現在の規模では採用しない。

## Consequences

- SQLite固有のsnapshotとロックを、同一ホスト内で一貫して検証できる
- インフラ構成と運用手順を単純に保てる
- ホスト障害中はサービスを継続できず、水平方向のスケールアウトもできない
- 永続ディスク、空き容量、WAL、バックアップ、復旧手順の運用が必須となる
- Cloud Run、複数ホスト、外部DBへ移行する場合は、新しいADRで本決定をsupersedeする
- 移行時は同期CSVを無効化し、ドライバ別に再検証するか、非同期生成とオブジェクトストレージ配信へ再設計する
