## 詳細

### C-1: タッチボタンの二重発火（Critical）

- 場所: `index.html:857-863` (`setupTouchEvent`)
- 内容: 各ボタンに `pointerdown` と `click` の両方を登録している。Pointer Events 仕様では `pointerdown` 内の `preventDefault()` は互換マウスイベント (mousedown / mouseup) を抑止するが **`click` は抑止しない**。そのため 1 タップ（クリック）でハンドラが 2 回実行される。
- 影響:
  - 「左 / 右」: 1 タップで 2 マス移動
  - 「回転」: 1 タップで 180° 回転
  - 「落下」: 1 タップで 2 マス落下
  - 「一時停止」: 2 回トグルされ **実質無効**（押しても止まらないように見える）
  - 「ハードドロップ」: 2 回目が no-op のため実害なし
- 対処案: 登録イベントを一本化する。`window.PointerEvent` が利用可能なら `pointerdown` のみ、非対応環境でのみ `click` にフォールバックする。

```js
const eventName = window.PointerEvent ? 'pointerdown' : 'click';
btn.addEventListener(eventName, handler);
```

### R-1: キーリピート未対策（要対応）

- 場所: `index.html:745-776` (keydown ハンドラ)
- 内容: plan.md ステップ 13 に「キーリピート配慮、誤入力抑制」とあるが、`e.repeat` を無視していないため OS のキーリピートで単発アクションが連発する。
  - `↑` 長押し: ブロックが高速で回転し続ける
  - `Space` 長押し: ハードドロップが連発し、着地後すぐ次のブロックも落下する
  - `P` 長押し: 一時停止が高速でトグルされ画面が点滅する
- 対処案: keydown ハンドラの先頭付近で単発アクション（回転・ハードドロップ・一時停止・リスタート）に対して `if (e.repeat) return;` を適用する。左右移動・ソフトドロップは `keys` テーブル + `handleInput()` の自前リピート制御を使っているため影響なし。

### R-2: タッチ環境にリスタート導線がない（要対応）

- 場所: `index.html:752-757`（リスタートは `Enter` キーのみ）
- 内容: plan.md ステップ 16 は「一時停止・再開・リスタート導線」を要件としているが、ゲームオーバー後の再開手段はキーボードの `Enter` のみ。タッチ端末では画面リロードするしかない（README にも「Enter キーか画面リロードで再開」と記載）。
- 対処案: ゲームオーバー時に「一時停止」ボタンのラベルを「🔁 リスタート」に切り替え、押下時に `init()` を呼ぶ。`updateUI()` が状態に応じてラベルを更新すれば実装は小さい。README の該当記述も更新する。

### S-1: Google Fonts 外部依存と「依存なし」記述の矛盾（不整合）

- 場所: `index.html:7-10`（Google Fonts の `<link>`）
- 内容: plan.md の Decisions は「外部ライブラリなし」、README は「追加の依存関係やビルド手順はありません」と記載しているが、実装は Google Fonts (Inter / Orbitron) をネットワーク経由で読み込む。オフラインでもフォールバックフォントで動作はするが、見た目が変わり、記述と実態がずれている。
- 対処案: 次のいずれか。
  1. システムフォントスタックに置き換えて完全オフライン化する（「単一ファイル完結」の decision に最も忠実）
  2. README に「フォントのみ Google Fonts を利用（オフライン時はフォールバック）」と注記する（変更最小）

### S-2: ハードドロップが即時固定でない（不整合・非標準）

- 場所: `index.html:771-775`（キーボード）、`index.html:839-846`（タッチ）
- 内容: ハードドロップは最下段まで移動するだけで、固定は次の落下更新（レベル 1 では最大 800ms 後）に行われる。その間は移動・回転が可能で、標準的なテトリスの挙動（即時固定）と異なる。README には現挙動が明記されているため「文書化された仕様」ではあるが、操作感として意図しないスライドが起きやすい。
- 対処案（推奨・合意済み）: ハードドロップ直後に固定処理（`fixBlock` → `clearLines` → 次ブロック出現 → ゲームオーバー判定）まで実行するよう変更し、README の補足記述を更新する。実装時は `update()` 内の固定処理を関数として切り出し、ハードドロップから共用するのがよい。

### S-3: 開始 UI がない（不整合・軽微）

- 場所: plan.md:18（ステップ 6「開始/再開 UI」）
- 内容: 実装はページ読み込みと同時に自動開始する。一時停止・再開はあるため実害は小さいが、plan.md の記述との差分。
- 対処案: 仕様側（plan.md / README）を実態に合わせて「自動開始」と追記するか、タイトル画面（タップ / キーで開始）を追加する。前者で十分。

### N-1: window blur 時のキー押下状態の残留（提案）

- 場所: `index.html:740-743, 778-780`（`keys` テーブルと keyup ハンドラ）
- 内容: キーを押したままウィンドウのフォーカスが外れると `keyup` が発火せず、`keys` に押下状態が残る。復帰後、キーを押していないのにブロックが動き続ける可能性がある。
- 対処案: `window` の `blur` イベントで `keys` をクリアする。あわせて自動一時停止にするとプレイ体験としても自然。

### N-2: Canvas 内部解像度が低く拡大表示が粗い（提案）

- 場所: `index.html:342`（`width="200" height="400"`）、`index.html:94`（`image-rendering: pixelated`）
- 内容: 内部解像度 200×400 の Canvas を CSS で最大 55vh まで拡大しており、デスクトップでは約 3 倍に引き伸ばされる。`pixelated` によりぼやけはしないが、README が謳う「光沢加工された立体的なブロック」のグラデーションが粗く見える。
- 対処案: `devicePixelRatio` と表示サイズに応じて `canvas.width / height` を設定し、描画スケールを `ctx.scale()` で補正する。`pixelated` は不要になる。

### N-3: updateUI() が毎フレーム DOM を書き換える（Nit）

- 場所: `index.html:874-884`（`gameLoop` から毎フレーム `updateUI()` を呼び出し）
- 内容: スコア等が変化していなくても毎秒約 60 回 `textContent` と `classList` を更新している。実害はほぼないが無駄。
- 対処案: スコア・レベル・状態が変化したときのみ `updateUI()` を呼ぶ（固定・ライン消去・ポーズ切替・ゲームオーバー時）。

### N-4: 命名・定数配置の乱れ（Nit）

- 場所: `index.html:408`（グローバル `currentTime`）、`index.html:874`（`gameLoop` の引数 `currentTimeStamp`）、`index.html:785-786`（`moveDelay` / `softDropDelay`）
- 内容:
  - グローバル `currentTime`（累積プレイ時間）と rAF タイムスタンプ `currentTimeStamp` が紛らわしい。
  - `moveDelay` / `softDropDelay` が `handleInput()` 内のローカル定義で、他のゲーム定数（`INITIAL_DROP_INTERVAL` 等）はファイル先頭にあり配置が不統一。
- 対処案: `currentTime` を `elapsedTime` 等に改名し、入力ディレイ定数をファイル先頭の定数群へ移動する。

### N-5: ロジックと I/O が未分離（提案・構造）

- 場所: `index.html:377-889`（script 全体）
- 内容: 盤面・スコア等がすべてグローバル変数で、ロジック関数（`canMove` / `rotate` / `clearLines` 等）がそれらに直接依存している。Node 上でのユニットテストはそのままでは不可能。ただしクラシックスクリプトのため各関数・変数は `window` に露出しており、**ブラウザ内からは自由に呼び出し・状態注入ができる**。
- 対処案: 「単一 HTML 完結」の決定を維持するため分割はせず、Playwright の `page.evaluate` からロジックを直接検証する方針とする（詳細は `docs/test-plan.md`）。将来的に大規模化する場合のみ `game.js` 分離を再検討する。

## 正しく実装されていることを確認した項目

- `canMove`: 壁・床・既存ブロックとの衝突判定（盤面上端より上のセルは正しく許容）
- `getRandomBlock`: 7 バッグ方式（Fisher–Yates シャッフル、`init()` でのバッグリセット含む）
- `clearLines`: 複数ライン同時消去、`splice` + `unshift` による詰め直し、スコア表 100 / 300 / 500 / 800 × レベル
- レベル進行: `floor(linesCleared / 10) + 1`、閾値をまたぐ複数ライン消去でも正しく昇格
- `rotate`: 転置 + 反転による回転と壁蹴り（失敗時は形状・位置を復元）
- 一時停止中は経過時間が加算されず、再開後に落下タイミングがずれない
- ゲームオーバー判定と `Enter` によるリスタート（全状態のリセットを確認）
- セキュリティ: ユーザー入力の HTML 挿入はなく（`textContent` のみ使用）、XSS の懸念なし
