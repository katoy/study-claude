## 1. テスト計画と実行方針

### 1.1 テスト構成
本アプリは以下の3つのレイヤーでテストを実施し、カバレッジ100%（自作コード部分）を担保します。

1. **Unit Test (単体テスト)**: `Vitest` + `jsdom`
   - ビジネスロジック、データ変換、サニタイズ、データ永続化の純粋関数を検証します。
2. **Integration Test (統合テスト / UIテスト)**: `Vitest` + `jsdom` + `@testing-library/dom`
   - メイン画面および詳細画面のDOM操作、ユーザーインタラクション、モーダル制御などをシミュレートし検証します。
3. **E2E / 手動動作確認シナリオ**:
   - 最終ビルドされた単一HTMLファイル (`dist/index.html`) に対し、ブラウザ上でオフライン動作やリロード永続化、XSS対策の堅牢性を検証します。

### 1.2 テスト実行コマンド
```bash
npm test                  # 単発テスト実行
npm run test:coverage     # カバレッジ計測 (Statements/Branches/Functions/Lines 100% 必須)
```

## 2. Unit Test (単体テスト) ケース

### 2.1 ToDoモデル (`src/models/todo.js`)
*   **UT-MOD-001: 新規ToDoオブジェクト生成 (`createTodo`)**
    *   **入力値**: タイトル「  買出しに行く  」(前後にスペース)、詳細「牛乳」、日時「なし」
    *   **期待される結果**:
        *   `id` が `crypto.randomUUID()` 形式のUUID文字列であること。
        *   `title` が trim され「買出しに行く」となること。
        *   `dueType` が `'none'`、`dueAt` が `null` となること。
        *   `completed` が `false` であること。
        *   `createdAt` および `updatedAt` が同一の有効な ISO 8601 UTC文字列であること。
*   **UT-MOD-002: 既存ToDoオブジェクト更新 (`updateTodo`)**
    *   **入力値**: 既存ToDoオブジェクト、新しいタイトル「本を買う」、詳細「技術書」
    *   **期待される結果**:
        *   `id` および `createdAt` が元のオブジェクトから変更されないこと。
        *   `title` が「本を買う」に更新されること。
        *   `updatedAt` が現在時刻の UTC文字列に更新され、`createdAt` より後の時刻になること。
*   **UT-MOD-003: 防御的バリデータ (`isValidTodo`) - 正常値**
    *   **入力値**: 有効な全フィールドを持つToDoオブジェクト
    *   **期待される結果**: `true` を返すこと。
*   **UT-MOD-004: 防御的バリデータ (`isValidTodo`) - 異常値**
    *   **入力値**: `null`、オブジェクト以外、または以下の欠損・型違い
        *   `id` が欠損または非文字列
        *   `title` が欠損、非文字列、または空文字(trim後)
        *   `dueType` が `'none' | 'date' | 'datetime'` 以外
        *   `dueAt` が `'none'` なのに `null` 以外
        *   `completed` が非boolean
        *   `createdAt`/`updatedAt` が欠損または不正な日付文字列
    *   **期待される結果**: すべて `false` を返すこと。

### 2.2 バリデーションロジック (`src/logic/validation.js`)

    *   **テストケース**:

        *   1文字 -> 成功
        *   100文字 -> 成功

*   **UT-VAL-002: 詳細文字数検証 (Quillプレーンテキスト換算)**

        *   1文字 -> 成功

*   **UT-VAL-003: Quill特有の末尾改行文字 (`\n`) 処理検証**

        *   ユーザーが「あ」と入力（`quill.getText()` が `"あ\n"` を返す）とき、文字数を「1文字」とカウントすること。

### 2.3 日付・時刻フォーマット・変換 (`src/date/dateFormat.js`)
*   **UT-DAT-001: 「日のみ」指定時のUTC変換**
    *   **入力値**: 日付文字列 `"2026-07-24"` (JST)
    *   **期待される結果**: その日のJST 23:59:59に相当するUTC ISO文字列 `"2026-07-24T14:59:59.000Z"` に変換されること。
*   **UT-DAT-002: 「時まで」指定時のUTC変換**
    *   **入力値**: 日時文字列 `"2026-07-24T09:00"` (JST)
    *   **期待される結果**: JST 09:00に相当するUTC ISO文字列 `"2026-07-24T00:00:00.000Z"` に変換されること。
*   **UT-DAT-003: 曜日を含む日付表示用フォーマット**
    *   **テストケース**:
        *   `dueType === 'date'` で `"2026-07-24T14:59:59.000Z"` (JST 2026-07-24 23:59:59)
            -> 期待値: `"2026-07-24 (金)"` (曜日は丸括弧で囲まれること)
        *   `dueType === 'datetime'` で `"2026-07-24T00:00:00.000Z"` (JST 2026-07-24 09:00)
            -> 期待値: `"2026-07-24 (金) 09:00"`
        *   `dueType === 'none'` の場合
            -> 期待値: `"—"` (ダッシュ)
*   **UT-DAT-004: 日付差の計算 (`daysBetween`)**
    *   **テストケース**:
        *   `"2026-07-24"` と `"2026-07-24"` -> 期待値: `0` (同日)
        *   `"2026-07-24"` と `"2026-07-25"` -> 期待値: `1` (翌日)
        *   `"2026-07-24"` と `"2026-07-23"` -> 期待値: `-1` (前日・期限超過)
    *   **重要: JST ベース計算の検証 ⚠️**
        *   UTC ISO 文字列を入力として受け取り、**JST 日付に変換してから**日数差を計算すること。
        *   例：
            *   入力: `"2026-07-24T14:59:59.000Z"` (JST 2026-07-24 23:59:59) と `"2026-07-25T14:59:59.000Z"` (JST 2026-07-25 23:59:59)
            *   計算: JST 2026-07-24 と JST 2026-07-25 の差 = 1日
            *   **バグの例**: UTC ベース計算を行うと、同じ UTC 日付（2026-07-24）だけで判定してしまい、日数差が 0 になってしまう。
        *   タイムゾーン境界や微細なミリ秒差に左右されず、常に JST 日付で正確に計算されていることを検証する。

### 2.4 セクション分類ロジック (`src/logic/sections.js`)
※ `vi.setSystemTime()` を用い、基準現在時刻を `2026-07-24T10:00:00Z` (JST 19:00:00) に固定して実施。
*   **UT-SEC-001: セクション判定分類**
    *   **テストケース**:
        1.  締切が JST 2026-07-24 中のタスク (`dueType='date'` / `"2026-07-24T14:59:59Z"`) -> 「本日中」セクション
        2.  締切が JST 2026-07-24 23:59 のタスク (`dueType='datetime'` / `"2026-07-24T14:59:00Z"`) -> 「本日中」セクション
        3.  締切が JST 2026-07-25 中のタスク (`dueType='date'` / `"2026-07-25T14:59:59Z"`) -> 「明日まで」セクション
        4.  締切が JST 2026-07-23 (昨日) のタスク (`dueType='date'` / `"2026-07-23T14:59:59Z"`) -> 「それ以外」セクション
        5.  締切が JST 2026-07-26 (明後日) のタスク (`dueType='date'` / `"2026-07-26T14:59:59Z"`) -> 「それ以外」セクション
        6.  締切未設定 (`dueType='none'`) のタスク -> 「それ以外」セクションの末尾
    *   **期待される結果**: 各ToDoが正しい分類に振り分けられること。

*   **UT-SEC-002: 空セクションの除外**
    *   **入力値**: 「本日中」と「それ以外」のToDoのみが存在し、「明日まで」のToDoが0件
    *   **期待される結果**: 出力されるセクション配列に「明日まで」セクション（および空配列のitems）が含まれないこと。

*   **UT-SEC-003: JST タイムゾーン境界での正しい分類（バグ回帰テスト）⚠️ 重要**
    *   **背景**: UTC と JST の日付が異なる場合、日付計算をUTC ベースで行うと誤分類が発生する。例えば、現在が JST 日付では「本日」だが UTC 日付では「昨日」という時刻帯がある（UTC 0:00～15:00 は JST では翌日）。
    *   **テストケース 1: UTC 15:00前後（JST 0:00前後）での分類**
        *   現在時刻: JST 2026-07-25 05:07:34 = UTC 2026-07-24 20:07:34 を基準に設定。
        *   タスク A: JST 2026-07-25 23:00 (UTC 2026-07-25 14:00) に期限 -> 「本日中」（JST 2026-07-25 中だから）
        *   タスク B: JST 2026-07-26 06:03 (UTC 2026-07-25 21:03) に期限 -> 「明日まで」（JST 2026-07-26 だから）✓
        *   タスク C: JST 2026-07-27 に期限 -> 「それ以外」
        *   **期待される結果**: ✓ タスク B が「明日まで」に正しく分類されること（UTC ベース計算では誤って「明日」になる可能性がある）。
    *   **テストケース 2: JST 00:00:00 の瞬間（UTC 15:00:00）**
        *   現在時刻: JST 2026-07-25 00:00:00 = UTC 2026-07-24 15:00:00 に設定。
        *   タスク X: JST 2026-07-25 00:00:01 (UTC 2026-07-24 15:00:01) に期限 -> 「本日中」
        *   タスク Y: JST 2026-07-26 00:00:00 (UTC 2026-07-25 15:00:00) に期限 -> 「明日まで」
        *   **期待される結果**: ✓ JST 日付で正確に分類されること。
    *   **実装上の注意**: `daysBetween()` は**常に JST 日付に変換して比較する**ため、UTC の日付境界に左右されずに正確に計算できる。
*   **UT-SRT-001: 同一セクション内でのソート基準**
    *   **テストケース**:
        1.  **締切あり同士**: `dueAt` が `"2026-07-24T09:00:00Z"` の項目と `"2026-07-24T05:00:00Z"` の項目。
            -> 期待値: 締切時間が早い（`05:00`）方が前にソートされること。
        2.  **締切なし同士**: `createdAt` が `"2026-07-24T01:00:00Z"` (古い) の項目と `"2026-07-24T02:00:00Z"` (新しい) の項目。
            -> 期待値: 作成日時が古い（`01:00`）方が前にソートされること。
        3.  **締切ありと締切なしの混在**:
            -> 期待値: 締切ありの項目が、作成日時に関わらず常に締切なしの項目より前に表示されること。

            -> 項目1, 項目2 共に `dueAt: "2026-07-24T09:00:00Z"`

## 3. Integration Test (統合・UIテスト) ケース

### 3.2 詳細画面 UI (`src/ui/detailView.js`)
*   **IT-DET-001: モーダルの起動モード制御**
    *   **テストケース**:
        1.  **新規登録モード**:
            -> 手順: メイン画面の「新規」ボタンをクリック。
            -> 期待値: `<dialog>` がモーダル表示され、タイトル・詳細・日付ピッカーが空（デフォルト）の状態であること。
        2.  **編集モード**:
            -> 手順: メイン画面の既存ToDoの「タイトル」をクリック。
            -> 期待値: `<dialog>` がモーダル表示され、該当ToDoのタイトル、日時選択ラジオボタンの選択状態、Quill内のHTML、が正確に初期値としてロードされていること。
*   **IT-DET-002: タイトル欄のバリデーションとカウンター**
    *   **手順**: タイトル欄に文字を入力する。
    *   **期待される結果**:
        *   入力するたびにカウンターが `n / 100` の形式でリアルタイムに増減すること。
        *   100文字を超えて入力しようとした場合、入力自体がブロックされる（キー入力を受け付けない）こと。

*   **IT-DET-003: 日時指定ラジオボタンとカレンダー UI 入力フォームの連動**
    *   **手順**: 日時指定ラジオボタンを切り替え、カレンダー UI から日付・時間を入力する。
    *   **期待される結果**:

        *   「日のみ」選択時: 日付用のカレンダー UI ピッカー (`<input type="date">` を重ねたカスタム入力欄) のみが表示・活性化され、入力された値およびプレースホルダーが `年-月-日`（`YYYY-MM-DD`）の書式で統一されてテキスト表示されること。
        *   「時まで」選択時: 日時用のカレンダー UI ピッカー (`<input type="datetime-local">` を重ねたカスタム入力欄) のみが表示・活性化され、入力された値およびプレースホルダーが `年-月-日 時:分`（`YYYY-MM-DD HH:mm`）の書式で統一されてテキスト表示されること。
*   **IT-DET-004: 詳細エディタ（Quill）の文字数制限とカウンター**
    *   **手順**: リッチエディタに文字を入力する（リッチテキストアダプター経由）。
    *   **期待される結果**:
        *   Quill内のプレーンテキスト（末尾の自動改行文字 `\n` を除外した文字数）に基づいて、カウンターが `n / 2000` でリアルタイムに増減すること。
        *   文字数が2000文字を超えた場合、保存ボタンが `disabled` 状態になること（または警告が表示されること）。
*   **IT-DET-005: 保存およびキャンセルの実行処理**
    *   **テストケース**:
        1.  **保存**:
            -> 手順: タイトルを入力し「保存」ボタンをクリック。
            -> 期待値: `localStorage` にデータが保存され、モーダルが閉じ、メイン画面のリストが再描画されて新規タスクが追加されていること。
        2.  **キャンセル**:
            -> 手順: タイトルを入力し「キャンセル」ボタンをクリック。
            -> 期待値: データは保存されず、モーダルが閉じ、メインリストも変更されないこと。
        3.  **ダイアログ外クリック**:
            -> 手順: モーダル背景領域をクリック。
            -> 期待値: キャンセルボタンと同様に、保存されずにモーダルが閉じること。

## 4. E2E / 手動動作確認シナリオ (最終成果物検証)

ビルド成果物である `dist/index.html` 単一ファイルに対し、実際のブラウザで以下のシナリオに沿って動作確認を行います。

### 4.1 シナリオ1: オフライン完全動作テスト
1.  **手順**:
    *   `npm run build` を実行し、`dist/index.html` を生成する。
    *   Chrome などのブラウザのデベロッパーツールを開き、「ネットワーク」タブで **「オフライン (Offline)」** モードに設定する。
    *   `dist/index.html` をブラウザで直接開く (`file://` スキーム)。
2.  **検証項目**:
    *   リッチエディタ（Quill）のアイコンやスタイルが崩れず、正しく表示・入力・装飾できること（外部CDNからのCSS/JS読み込みエラーが発生していないこと）。
    *   タスクの新規登録、詳細編集、完了トグル、一括削除がすべてエラーなくスムーズに実行できること。

### 4.2 シナリオ2: データの永続化とリロードテスト
1.  **手順**:
    *   ブラウザでToDoアプリを開く。
    *   「明日まで」のタスクを1件登録する。
    *   「本日中」のタスクを1件登録し、それを「完了」状態にする。
    *   ブラウザの「再読み込み (F5)」を実行する。
2.  **検証項目**:
    *   リロード後も、登録した2件のタスクが正しいセクション（本日中、明日まで）に表示されていること。
    *   「本日中」のタスクが「完了済み」状態（取り消し線、グレーアウト、ボタンが「未完了」）を維持していること。

### 4.3 シナリオ3: セキュリティ (XSS) 脆弱性テスト
1.  **手順**:
    *   新規ToDoを登録する。
    *   タイトル欄に以下を入力:
        `正常なタイトル <script>alert('XSS Title')</script>`
    *   詳細（リッチエディタ）にツールバー経由ではなく、直接HTMLペースト等の手段を想定し、以下の文字列の保存を試みる（またはAPI/リポジトリへの直接書き込みを模倣）:
        `正常な詳細 <img src="x" onerror="alert('XSS Detail')">`
    *   「保存」し、メイン画面の一覧で表示を確認する。さらに、そのタスクをクリックして編集モーダルで再表示する。
2.  **検証項目**:
    *   ブラウザ上で `alert` ダイアログが表示されないこと。
    *   開発者ツールのDOMツリーを確認し、危険なタグや属性（`script`, `onerror`）が除去されていること。

### 4.4 シナリオ4: カレンダー UI による日付/時間入力テスト
1.  **手順**:
    *   アプリを開き、新規登録ボタンをクリックする。
    *   日時指定で「日のみ指定」を選択する。
    *   日付ピッカーの右側にあるカレンダーアイコンをクリックするか、入力エリアにフォーカスし、ブラウザ標準のカレンダー UI ポップアップが展開されることを確認する。
    *   カレンダー上で特定の日付をクリックして選択する。
    *   同様に「時間と分も指定」を選択し、日時ピッカーからカレンダー UI および時刻ピッカーで日時を選択・設定する。
2.  **検証項目**:
    *   キーボード入力を行わずとも、カレンダー UI のマウスクリック（またはタップ）操作のみで日付と時間が正しく入力欄にセットされること。
    *   画面に表示されるプレースホルダーおよび入力済みの値が、ブラウザやOSの言語設定に関わらず常に `年-月-日` (`YYYY-MM-DD`) または `年-月-日 時:分` (`YYYY-MM-DD HH:mm`) の統一されたハイフン区切り書式で表示されること。
    *   設定した日時がそのまま正しく保存され、一覧画面の締切欄に反映されること。
